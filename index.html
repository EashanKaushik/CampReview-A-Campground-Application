YelpCamp application

See the application live!

Yelp Camp application created as a part of The web developer bootcamp course on Udemy. Basic functionalities were created as a part of the course, though there are a lot of them, that were implemented solely by me or using some of the additional tutorials uploadad on course's page.

The application follows REST paradigm, RWD techniques and was created using MEN stack, with EJS as a view engine.

The idea behind this app, was to create a web portal that will allow users to add campgrounds, comment them, rate and basically interact with them in any other way.

My goal, when creating it, was to showcase my abilities as a web-developer, both in fields like design + functionalities and back-end logic + data manipulation.

List of all resources used in this project (frameworks, libraries, modules and others) can be found at the bottom.

Feel free to visit my app and interact with it to see how it works. My suggestions:

  • Try to create an account (sign up)
  • Try to create a campground with/without specified location and see what happens when you do/don't specify it
  • Visit you own profile page
  • Visit other user's profile page
  • Rate a campground, then visit it to see how and if rating updates
  • Comment a campground
  • Try to visit http://yelp-camp-app-jw.herokuapp.com/campgrounds/new without logging in to your account, and see what happens
  • Try to sort campgrounds on /campgrounds page and see if it works

Please, keep in mind that:

1 . If the app is not live, then I'm probably making some changes to it and will leave it working once I'm done. If you want, try to revisit the URL in a couple of hours or so.

2 . If you want to sign up to my app to test this specific functionality, I advise to use fake data, since I'm not taking responsibility for potential data loss, in case you decided to sign up with your real email/name or a password that is valid in some online services. Furthermore, your input data is not going to be used anywhere else than in the app itself. Your data is not going to be stored in my database for longer than ~1h - that's how long approximately the server will be online. After this long, it goes to "sleep mode", once it resets when somebody tries to connect to it, a script launches that wipes out all the current data from the database and feeds it with random generated one. But just in case - please, don't use any of your real personal data

1. Data models

My app includes data models for campgrounds, comments and users. Models were created using mongoose.model() method on Schemas.

Diagram with DB relationships:

Campgrounds

var campgroundSchema = new mongoose.Schema({
        author: {
                id: { type: mongoose.Schema.Types.ObjectId, ref: "User" },
                username: String
        },
        image: { type: String, default: "/placeholder.jpg" },
        name: String,
        desc: String,
        price: Number,
        location: { type: String, default: "" },
        lat: { type: Number, default: null },
        lng: { type: Number, default: null },
        createdAt: { type: Date, default: Date.now },
        comments: [
                { type: mongoose.Schema.Types.ObjectId, ref: "Comment" }   
        ],
        isPending: { type: Boolean, default: false },
        rating: {
                votes: {
                        "5": { type: Number, default: 0 },
                        "4": { type: Number, default: 0 },
                        "3": { type: Number, default: 0 },
                        "2": { type: Number, default: 0 },
                        "1": { type: Number, default: 0 }           
                },
                averageRating: { type: Number, default: 0 },
                votesQuantity: { type: Number, default: 0 },
                multipliedRatingsSum: { type: Number, default: 0 },
        }
});

The fields are quite self explanatory, but here's description of some of them.

location - necessarry for Google Maps API. Along with lat and lng that stand for latitude and longitude.

author - it stores author's ID in form of mongoose.Schema.Types.ObjectId that references specific User model.

comments - array that stores IDs that reference specific Comment models.

rating - here you can see my approach to ratings - object, with possible rating values as keys. Value of each key is equal to quantity of votes that are equal to the key itself. For example, rating.votes["5"] = 5 means that now, there are five "5 star" ratings.

isPending - if you specify campground's location when creating it, a request should be sent to google to geocode your input. Considering billing behind Google Maps API, I can't allow people to make those requests however and whenever they want.if(location) then isPending is set to true, which means campground is not published until admin approves it. Only then, the location gets geocoded, and the map on /campgrounds INDEX page or /campgrounds/:id SHOW page, can be generated.

Users

var UserSchema = new mongoose.Schema({
        username: String,
        password: String,
        firstName: String,
        lastName: String,
        email: String,
        avatar: { type: String, default: "/placeholder-user.png" },
        createdAt: { type: Date, default: Date.now },
        isAdmin: { type: Boolean, default: false },
        ratings: [
                {
                        name: String,
                        value: Number
                        _id: { type: mongoose.Schema.Types.ObjectId, ref: "Campground" },

password - type is String though it's not stored "as is". It gets encrypted by passport. So this filed means, that you also have hash and salt.

ratings - campgrounds this user has rated, and what value he has given to each one

2. Routes

The application follows REST paradigm, but there are some custom routes I created and some different solutions here and there.

Campgrounds

URL HTTP verb Action
/campgrounds GET INDEX
/campgrounds/new GET NEW
/campgrounds/:id/edit GET EDIT
/campgrounds/campgrounds/:id GET SHOW
/campgrounds POST CREATE
/campgrounds/rate PUT custom route (rating campgrounds)
/campgrounds/:id PUT UPDATE
/campgrounds/pending/:id/publish PUT custom route (publishing pending campgrounds)
/campgrounds/:id DELETE DESTROY

Comments

URL HTTP verb Action
/campgrounds/:id/comments/new GET NEW (not in use now)
/campgrounds/:id/comments/:comment_id/edit GET EDIT (not in use now)
/campgrounds/:id/comments POST CREATE
/campgrounds/:id/comments/:comment_id PUT UPDATE
/campgrounds/:id/comments/:comment_id DELETE DESTROY

The reason why I don't have INDEX or SHOW route for comments, is that I don't have a need to display them all at once, or as an "object", anywhere, to anyone on the site, unless it's a campground page. Though admin has a functionality called admin panel, that I will describe later. It allows him to see all the comments, and edit/delete any of them.

General

URL HTTP verb Action
/ GET custom route (landing page)

Admin

URL HTTP verb Action
/admin GET custom route (show admin panel)

4. Functionalities

Campgrounds

  • creating campgrounds

    • if you specify a valid location (i.e. the one that actually exists), after admin approval it will get geocoded and a Google Map will pop up on your campground's SHOW page

    • you can upload a picture of your campground (achieved using multer and cloudinary)

  • editing campgrounds

    • if you are the owner of a campground or an admin, you will have acces to campground's edit form from various places - from the admin panel, campgrounds INDEX page, campgroud's SHOW page, user's SHOW page

    • you can upload a new picture when editing a campground

    • here are some rules regarding changing campground's location:

User role Is location specified in the edit form? Is location different than the previous one? Action
Regular user No - Update campground
Regular user Yes No Update campground
Regular user Yes Yes Update campground, don't publish it, set it to "pending" which means it will wait for admin approval
Regular user No Yes Update campground (That's the case of just removing campground's location on update)
Admin No - Update campground
Admin Yes No Update campground
Admin Yes Yes Update campground, make a request for geocoding specified location, publish campground
Admin No Yes Update campground (That’s the case of just removing campground’s location on update)

  • deleting campgrounds

    • you can delete campgrounds from various places if below condition is met - be it INDEX page, campground's SHOW page, user's SHOW page or admin panel.

    • you can delete a campground only if you own it or you are an admin.

    • anytime campground is deleted, comments associated with it get deleted too.

  • displaying campgrounds

    • on the INDEX page you will see Bootstrap 4 pill badges on campgrounds images, providing some information about a specific campground - it's location, average rating and price per night.

    • on the INDEX page - I implemented pagination of search results and sorting functionality. Sorting is preserved (when chosen) when changing page - achieved via query strings.

    • on the INDEX and SHOW page - if you own a campground, or you are an admin, you will see buttons allowing you to edit a campground or delete it.

    • on INDEX page - any user is allowed to rate a campground. Thi functionality was implemented with RateYo - in it's current state, this function makes an AJAX request to the database to update campground's rating. Then it increases average rating value on the pill associated with this campground. Though, if you have rated a campground before, then the stars will be read-only, showing your rating.

    • on INDEX page - you will see a 'Map' tab. If location is specified, then after clicking this tab a Google Map will be generated. After generating, this element recieves generated class, indicating that on the next click, map won't be generated. It means, that we won't be making useless requests to Google Maps API.

Comments

  • adding comments

    • happens on campground's SHOW page. You need to be logged in to do that.

  • editing comments

    • happens on campground's SHOW page too. Same condition applies.

  • deleting commens

    • you can delete a comment only if you own it or you are an admin. You can do that from various places - campground's SHOW page, user's SHOW page or admin panel.

  • displaying comments

    • comments show up on campground's SHOW page, and also in admin panel and specific user's SHOW page. In the last scenario I mean only comments associated with this exact user.

Users

  • adding users

    • it happens through signing up

    • you can upload your avatar picture

  • deleting user

    • only admin can delete users. It happens on admin panel.

    • when a user is deleted, all the comments and campgrounds he created get deleted too.

  • displaying user

    • user's have their own SHOW page, that displays all the campground they have created, all the comments and all the ratings they have given. Also, avatar picture, some basic personal data and date joined.

Admin

  • general

    • can make campgrounds with specified location without waiting for any kind of approval - after validating if input location is ok, it gets geocoded, saved in the database and Google Maps can be generated throughout the app

    • can edit/delete pretty much anything that exists in the database/you see in the app

    • can make other users admins or take this privilage away from them

    • has access to admin panel, which is basically a library of all campgrounds, comments and users currently in database
